## <a name="the-domain-entity-pattern"></a><span data-ttu-id="95c96-109">Il modello di entità di dominio</span><span class="sxs-lookup"><span data-stu-id="95c96-109">The Domain Entity pattern</span></span>

<span data-ttu-id="95c96-110">Le entità rappresentano gli oggetti di dominio e vengono definite principalmente da identità, la continuità e persistenza nel tempo e non solo per gli attributi che costituiscono le.</span><span class="sxs-lookup"><span data-stu-id="95c96-110">Entities represent domain objects and are primarily defined by their identity, continuity, and persistence over time, and not only by the attributes that comprise them.</span></span> <span data-ttu-id="95c96-111">Come viene visualizzato, Eric Evans "di un oggetto definito principalmente tramite l'identità viene chiamato un'entità."</span><span class="sxs-lookup"><span data-stu-id="95c96-111">As Eric Evans says, “an object primarily defined by its identity is called an Entity.”</span></span> <span data-ttu-id="95c96-112">Le entità sono molto importanti per il modello di dominio, poiché sono la base per un modello.</span><span class="sxs-lookup"><span data-stu-id="95c96-112">Entities are very important in the domain model, since they are the base for a model.</span></span> <span data-ttu-id="95c96-113">Pertanto, identificare e progettare con attenzione.</span><span class="sxs-lookup"><span data-stu-id="95c96-113">Therefore, you should identify and design them carefully.</span></span>

<span data-ttu-id="95c96-114">*Identità di un'entità può attraversare più microservizi o contesti limitato.*</span><span class="sxs-lookup"><span data-stu-id="95c96-114">*An entity’s identity can cross multiple microservices or Bounded Contexts.*</span></span>

<span data-ttu-id="95c96-115">La stessa identità (anche se non è la stessa entità) possono essere modellate in più contesti delimitata o microservizi.</span><span class="sxs-lookup"><span data-stu-id="95c96-115">The same identity (though not the same entity) can be modeled across multiple Bounded Contexts or microservices.</span></span> <span data-ttu-id="95c96-116">Tuttavia, non implica che la stessa entità, con gli stessi attributi e logica potrebbe essere implementate in più contesti delimitata.</span><span class="sxs-lookup"><span data-stu-id="95c96-116">However, that does not imply that the same entity, with the same attributes and logic would be implemented in multiple Bounded Contexts.</span></span> <span data-ttu-id="95c96-117">Al contrario, le entità in ogni contesto delimitata limitare i relativi attributi e i comportamenti di quelli richiesti nel dominio del contesto che delimitata.</span><span class="sxs-lookup"><span data-stu-id="95c96-117">Instead, entities in each Bounded Context limit their attributes and behaviors to those required in that Bounded Context’s domain.</span></span>

<span data-ttu-id="95c96-118">Ad esempio, l'entità dell'acquirente potrebbe essere la maggior parte degli attributi di una persona che sono definiti nell'entità in microservizio il profilo o l'identità, tra cui l'identità utente.</span><span class="sxs-lookup"><span data-stu-id="95c96-118">For instance, the buyer entity might have most of a person’s attributes that are defined in the user entity in the profile or identity microservice, including the identity.</span></span> <span data-ttu-id="95c96-119">L'entità buyer nell'ordinamento microservizio potrebbe invece essere meno attributi, perché solo determinati dati buyer sono correlati al processo di ordine.</span><span class="sxs-lookup"><span data-stu-id="95c96-119">But the buyer entity in the ordering microservice might have fewer attributes, because only certain buyer data is related to the order process.</span></span> <span data-ttu-id="95c96-120">Il contesto di ogni microservizio o delimitata contesto ha un impatto relativo modello di dominio.</span><span class="sxs-lookup"><span data-stu-id="95c96-120">The context of each microservice or Bounded Context impacts its domain model.</span></span>

<span data-ttu-id="95c96-121">*Entità di dominio deve implementare il comportamento oltre all'implementazione di attributi dei dati*</span><span class="sxs-lookup"><span data-stu-id="95c96-121">*Domain entities must implement behavior in addition to implementing data attributes*</span></span>

<span data-ttu-id="95c96-132">Nel suo post [AnemicDomainModel](https://martinfowler.com/bliki/AnemicDomainModel.html), Martin Fowler descrive un modello di dominio anemic in questo modo:</span><span class="sxs-lookup"><span data-stu-id="95c96-132">In his post [AnemicDomainModel](https://martinfowler.com/bliki/AnemicDomainModel.html), Martin Fowler describes an anemic domain model this way:</span></span>

<span data-ttu-id="95c96-133">Il sintomo di un modello di dominio Anemic base è inizialmente sembra reali.</span><span class="sxs-lookup"><span data-stu-id="95c96-133">The basic symptom of an Anemic Domain Model is that at first blush it looks like the real thing.</span></span> <span data-ttu-id="95c96-134">Sono presenti oggetti, molti denominato dopo i nomi nello spazio di dominio e questi oggetti sono connessi con le relazioni avanzate e la struttura con i modelli di dominio true.</span><span class="sxs-lookup"><span data-stu-id="95c96-134">There are objects, many named after the nouns in the domain space, and these objects are connected with the rich relationships and structure that true domain models have.</span></span> <span data-ttu-id="95c96-135">Catch viene fornito quando si osserva il comportamento e ci si accorge che è praticamente qualsiasi comportamento di tali oggetti, rendendoli leggermente più contenitori di getter e Setter.</span><span class="sxs-lookup"><span data-stu-id="95c96-135">The catch comes when you look at the behavior, and you realize that there is hardly any behavior on these objects, making them little more than bags of getters and setters.</span></span>

<span data-ttu-id="95c96-136">Naturalmente, quando si utilizza un modello di dominio anemic, i modelli di dati verranno utilizzati da un set di oggetti servizio (denominata in genere il *livello business*) cui acquisire tutta la logica di business o di dominio.</span><span class="sxs-lookup"><span data-stu-id="95c96-136">Of course, when you use an anemic domain model, those data models will be used from a set of service objects (traditionally named the *business layer*) which capture all the domain or business logic.</span></span> <span data-ttu-id="95c96-137">Il livello business si trova nella parte superiore del modello di dati e utilizza il modello di dati come dati.</span><span class="sxs-lookup"><span data-stu-id="95c96-137">The business layer sits on top of the data model and uses the data model just as data.</span></span>

<span data-ttu-id="95c96-138">Il modello di dominio anemic è una progettazione di stile procedurale.</span><span class="sxs-lookup"><span data-stu-id="95c96-138">The anemic domain model is just a procedural style design.</span></span> <span data-ttu-id="95c96-139">Oggetti entità anemic non sono oggetti reali perché dispongono di un comportamento (metodi).</span><span class="sxs-lookup"><span data-stu-id="95c96-139">Anemic entity objects are not real objects because they lack behavior (methods).</span></span> <span data-ttu-id="95c96-140">Contengono solo le proprietà di dati e non è pertanto progettazione orientata agli oggetti.</span><span class="sxs-lookup"><span data-stu-id="95c96-140">They only hold data properties and thus it is not object-oriented design.</span></span> <span data-ttu-id="95c96-141">Inserendo tutti i comportamenti out in oggetti del servizio (il livello business) essenzialmente finire con [assai complicato](https://en.wikipedia.org/wiki/Spaghetti_code) o [script transazione](https://martinfowler.com/eaaCatalog/transactionScript.html), e pertanto persi i vantaggi che un modello di dominio fornisce.</span><span class="sxs-lookup"><span data-stu-id="95c96-141">By putting all the behavior out into service objects (the business layer) you essentially end up with [spaghetti code](https://en.wikipedia.org/wiki/Spaghetti_code) or [transaction scripts](https://martinfowler.com/eaaCatalog/transactionScript.html), and therefore you lose the advantages that a domain model provides.</span></span>

<span data-ttu-id="95c96-142">Indipendentemente dal fatto che, se lo microservizio o il contesto limitato è molto semplice (un servizio CRUD), il modello di dominio anemic sotto forma di oggetti entità con solo le proprietà di dati potrebbe essere sufficiente e che non sia opportuno implementare modelli DDD più complessi.</span><span class="sxs-lookup"><span data-stu-id="95c96-142">Regardless, if your microservice or Bounded Context is very simple (a CRUD service), the anemic domain model in the form of entity objects with just data properties might be good enough, and it might not be worth implementing more complex DDD patterns.</span></span> <span data-ttu-id="95c96-143">In tal caso, sarà sufficiente un modello di persistenza, poiché è stato creato intenzionalmente un'entità solo i dati per scopi CRUD.</span><span class="sxs-lookup"><span data-stu-id="95c96-143">In that case, it will be simply a persistence model, because you have intentionally created an entity with only data for CRUD purposes.</span></span>

<span data-ttu-id="95c96-144">Che è il motivo per cui le architetture di microservizi sono ideali per un approccio architetturale più a seconda del contesto ogni delimitata.</span><span class="sxs-lookup"><span data-stu-id="95c96-144">That is why microservices architectures are perfect for a multi-architectural approach depending on each Bounded Context.</span></span> <span data-ttu-id="95c96-145">Ad esempio, in eShopOnContainers, l'ordinamento microservizio implementa modelli DDD, ma non microservizio il catalogo, che è un semplice servizio CRUD.</span><span class="sxs-lookup"><span data-stu-id="95c96-145">For instance, in eShopOnContainers, the ordering microservice implements DDD patterns, but the catalog microservice, which is a simple CRUD service, does not.</span></span>

<span data-ttu-id="95c96-146">Alcune persone, si supponga che il modello di dominio anemic è un anti-pattern.</span><span class="sxs-lookup"><span data-stu-id="95c96-146">Some people say that the anemic domain model is an anti-pattern.</span></span> <span data-ttu-id="95c96-147">Questo dipende si sta implementando.</span><span class="sxs-lookup"><span data-stu-id="95c96-147">It really depends on what you are implementing.</span></span> <span data-ttu-id="95c96-148">Se il microservizio si sta creando è semplice sufficientemente (ad esempio, un servizio CRUD), al modello di dominio anemic non è un anti-pattern.</span><span class="sxs-lookup"><span data-stu-id="95c96-148">If the microservice you are creating is simple enough (for example, a CRUD service), following the anemic domain that microservice or Bounded Context.</span></span> <span data-ttu-id="95c96-150">In tal caso, impostandolo come potente modello con le entità che contengono dati e comportamento nonché implementare ### <a name="the-aggregate-pattern"></a><span data-ttu-id="95c96-178">Il modello di aggregazione</span><span class="sxs-lookup"><span data-stu-id="95c96-178">The Aggregate pattern</span></span>

<span data-ttu-id="95c96-179">Un modello di dominio contiene cluster delle entità di dati diversi e i processi che è possono controllare un'area significativa di funzionalità, ad esempio l'evasione dell'ordine o inventario.</span><span class="sxs-lookup"><span data-stu-id="95c96-179">A domain model contains clusters of different data entities and processes that can control a significant area of functionality, such as order fulfilment or inventory.</span></span> <span data-ttu-id="95c96-180">Un'unità DDD con granularità fine è l'aggregazione, che descrive un cluster o un gruppo di entità e i comportamenti che possono essere considerati come un'unità coesiva.</span><span class="sxs-lookup"><span data-stu-id="95c96-180">A more fine-grained DDD unit is the aggregate, which describes a cluster or group of entities and behaviors that can be treated as a cohesive unit.</span></span>

<span data-ttu-id="95c96-181">Si definisce in genere un'aggregazione in base alle transazioni di cui è necessario.</span><span class="sxs-lookup"><span data-stu-id="95c96-181">You usually define an aggregate based on the transactions that you need.</span></span> <span data-ttu-id="95c96-182">Un esempio classico è un ordine contenente un elenco di elementi di ordine.</span><span class="sxs-lookup"><span data-stu-id="95c96-182">A classic example is an order that also contains a list of order items.</span></span> <span data-ttu-id="95c96-183">Un elemento order in genere è un'entità.</span><span class="sxs-lookup"><span data-stu-id="95c96-183">An order item will usually be an entity.</span></span> <span data-ttu-id="95c96-184">Ma sarà un'entità figlio all'interno dell'aggregazione ordine, che conterrà anche l'entità ordine come relativa entità principale, in genere chiamato una radice di aggregazione.</span><span class="sxs-lookup"><span data-stu-id="95c96-184">But it will be a child entity within the order aggregate, which will also contain the order entity as its root entity, typically called an aggregate root.</span></span>

<span data-ttu-id="95c96-185">Identificazione di funzioni di aggregazione può essere difficile.</span><span class="sxs-lookup"><span data-stu-id="95c96-185">Identifying aggregates can be hard.</span></span> <span data-ttu-id="95c96-186">Un'aggregazione è un gruppo di oggetti che devono essere coerenti tra loro, ma non è possibile solo selezionare un gruppo di oggetti e contrassegnarli come un'aggregazione.</span><span class="sxs-lookup"><span data-stu-id="95c96-186">An aggregate is a group of objects that must be consistent together, but you cannot just pick a group of objects and label them an aggregate.</span></span> <span data-ttu-id="95c96-187">È necessario iniziare con il concetto di dominio e considerare le entità che vengono utilizzate nelle transazioni più comuni relativi a questo concetto.</span><span class="sxs-lookup"><span data-stu-id="95c96-187">You must start with a domain concept and think about the entities that are used in the most common transactions related to that concept.</span></span> <span data-ttu-id="95c96-188">Le entità che devono essere coerenti sono cosa costituisce un'aggregazione.</span><span class="sxs-lookup"><span data-stu-id="95c96-188">Those entities that need to be transactionally consistent are what forms an aggregate.</span></span> <span data-ttu-id="95c96-189">Considerare le operazioni di transazioni è probabilmente il modo migliore per identificare le aggregazioni.</span><span class="sxs-lookup"><span data-stu-id="95c96-189">Thinking about transaction operations is probably the best way to identify aggregates.</span></span>

### <a name="the-aggregate-root-or-root-entity-pattern"></a><span data-ttu-id="95c96-190">Il modello radice di aggregazione o entità radice</span><span class="sxs-lookup"><span data-stu-id="95c96-190">The Aggregate Root or Root Entity pattern</span></span>

<span data-ttu-id="95c96-191">Un'aggregazione è composta da almeno un'entità: la radice di aggregazione, definito anche come entità principale o primario ientity.</span><span class="sxs-lookup"><span data-stu-id="95c96-191">An aggregate is composed of at least one entity: the aggregate root, also called root entity or primary ientity.</span></span> <span data-ttu-id="95c96-192">Inoltre, può avere più entità figlio e gli oggetti di valore, con tutte le entità e oggetti che interagiscono per implementare le transazioni e il comportamento richiesto.</span><span class="sxs-lookup"><span data-stu-id="95c96-192">Additionally, it can have multiple child entities and value objects, with all entities and objects working together to implement required behavior and transactions.</span></span>

<span data-ttu-id="95c96-193">Lo scopo di una radice di aggregazione è garantire la coerenza della funzione di aggregazione; deve essere il solo punto di ingresso per l'aggregazione tramite i metodi di aggiornamenti o le operazioni di classe root.</span><span class="sxs-lookup"><span data-stu-id="95c96-193">The purpose of an aggregate root is to ensure the consistency of the aggregate; it should be the only entry point for updates to the aggregate through methods or operations in the aggregate root class.</span></span> <span data-ttu-id="95c96-194">È necessario apportare modifiche alle entità all'interno dell'aggregazione solo tramite la radice di aggregazione.</span><span class="sxs-lookup"><span data-stu-id="95c96-194">You should make changes to entities within the aggregate only via the aggregate root.</span></span> <span data-ttu-id="95c96-195">Si tratta di sorveglianza la coerenza dell'aggregazione, tenendo in considerazione tutte le invarianti e regole di uniformità che potrebbe essere necessario rispettare nell'aggregazione.</span><span class="sxs-lookup"><span data-stu-id="95c96-195">It is the aggregate’s consistency guardian, taking into account all the invariants and consistency rules you might need to comply with in your aggregate.</span></span> <span data-ttu-id="95c96-196">Se si modifica un oggetto di entità o un valore figlio in modo indipendente, la radice di aggregazione non è possibile verificare che la funzione di aggregazione sia in uno stato valido.</span><span class="sxs-lookup"><span data-stu-id="95c96-196">If you change a child entity or value object independently, the aggregate root cannot ensure that the aggregate is in a valid state.</span></span> <span data-ttu-id="95c96-197">Sarebbe ad esempio una tabella con un segmento separato.</span><span class="sxs-lookup"><span data-stu-id="95c96-197">It would be like a table with a loose leg.</span></span> <span data-ttu-id="95c96-198">Mantenere la coerenza è lo scopo principale della radice dell'aggregazione.</span><span class="sxs-lookup"><span data-stu-id="95c96-198">Maintaining consistency is the main purpose of the aggregate root.</span></span>
